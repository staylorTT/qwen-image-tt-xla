Qwen-Image 20B on 4x Blackhole -- Performance Analysis
========================================================
Date: 2023-03-23
Status: Working end-to-end, generating correct images


ARCHITECTURE OVERVIEW
---------------------
Model: Qwen-Image 20B (MMDiT / joint attention diffusion transformer)
  - 60 identical transformer blocks, each with:
    - AdaLN modulation (timestep conditioning via SiLU MLP)
    - Joint attention: image + text tokens share Q/K/V space
      - 6 linear projections (img Q/K/V, txt Q/K/V) -- column-parallel
      - QK RMSNorm
      - RoPE (rotary positional embeddings)
      - Scaled dot-product attention
      - 2 output projections (img, txt) -- row-parallel
    - 2 FFN paths (img, txt): GELU fc1 (col-par) -> fc2 (row-par)
    - Gated residual connections
  - Pre/post block ops: img_in, txt_in, txt_norm, time_text_embed, norm_out, proj_out
  - Text encoder: Qwen2.5-VL-7B (CPU)
  - VAE decoder: 3D causal conv (CPU)
  - Scheduler: flow-matching Euler (CPU)

Key dimensions:
  - Hidden dim: 3072 (24 heads x 128 head_dim)
  - FFN hidden: 8192 (GEGLU, so fc1 output is 16384 split in half)
  - Vocab: 3584 (text encoder output dim)
  - Latent channels: 16 -> packed to 64


TENSOR PARALLELISM (4 devices)
------------------------------
Column-parallel (shard output dim across chips):
  - QKV projections: [3072, 3072] -> [3072, 768] per chip
  - FFN fc1: [3072, 16384] -> [3072, 4096] per chip
  - Biases sharded to match

Row-parallel (shard input dim, all_reduce after):
  - Output projections: [3072, 3072] -> [768, 3072] per chip
  - FFN fc2: [8192, 3072] -> [2048, 3072] per chip
  - Biases replicated

Replicated (small, on all chips):
  - AdaLN MLP weights
  - QK-norm weights
  - RoPE tables
  - Swap permutation matrix


CURRENT IMPLEMENTATION
----------------------
100% ttnn (zero tt-lang custom kernels). All ops are stock ttnn builtins:
  ttnn.linear, ttnn.matmul, ttnn.layer_norm, ttnn.rms_norm, ttnn.silu,
  ttnn.transformer.scaled_dot_product_attention, ttnn.all_reduce,
  ttnn.multiply, ttnn.add, ttnn.concat, ttnn.slice, ttnn.reshape,
  ttnn.transpose

RoPE approach: permutation matrix
  - 128x128 matrix P that swaps adjacent elements
  - Applied AFTER QK-norm: q_swap = q @ P, then q_roped = q*cos + q_swap*sin_perm
  - Correct ordering (swap after norm, not before)
  - One small matmul per Q/K per block

AdaLN: concat-repeat expansion
  - temb (per-batch) expanded to fill TILE rows on CPU before upload
  - On device: linear -> concat([result]*n_repeat) -> slice into shift/scale/gate
  - Bug found and fixed: tile padding was zeroing rows 1-31 of temb,
    causing only every 32nd position to get correct modulation


PERFORMANCE RESULTS
-------------------
All measurements with 4x Blackhole, TP across 4 devices.
Weight preload is one-time ~170s (will be ~10s after extract_block_weights fix).

Resolution   Steps  SeqLen  Per-Step  Denoising  Notes
---------    -----  ------  --------  ---------  -----
256x256        4      256    0.26s      1.1s     Fast preview
256x256       20      256    0.26s      5.3s     Standard
512x512       60     1024    0.63s     37.7s     Good quality
1024x1024     60     4096    2.44s    146.6s     High res

Scaling: roughly linear in seq_len (attention-dominated).
256->512: 2.4x slower (4x seq), 512->1024: 3.9x slower (4x seq).
Sub-linear scaling suggests overhead from non-attention ops at smaller sizes.

WHAT IS ON CPU AND COULD BE MOVED TO DEVICE
--------------------------------------------

1. Pre-block ops (per step, currently ~5-20ms):
   - transformer.img_in(latents): Linear [64->3072], runs on CPU
   - transformer.txt_in(txt_norm(prompt_embeds)): LayerNorm + Linear [3584->3072]
   - transformer.time_text_embed(ts, hs): timestep MLP
   These are small and fast on CPU. Moving to device would eliminate one H2D
   transfer per step but adds complexity. Worth doing for tracing mode.

2. Post-block ops (per step, currently ~10-40ms):
   - from_tt (D2H transfer)
   - transformer.norm_out(hs, temb): AdaLN final norm
   - transformer.proj_out(hs): Linear [3072->64]
   Same as above -- small but required for tracing mode.

3. Scheduler step (per step, ~1-5ms):
   - Euler step: latents = latents + dt * noise_pred
   - Can be done on device trivially (just multiply + add)
   - The XLA demo.py already does this on device

4. Text encoder (~1s, one-time per prompt):
   - 7B VLM forward pass on CPU
   - Too large to fit on device alongside MMDiT weights
   - Could pre-compute and cache for repeated prompts
   - Not a bottleneck for interactive use

5. VAE decode (~3-10s, one-time per image):
   - 3D causal convolutions, complex architecture
   - Runs once per image, not per step
   - Significant wall-time but low priority for optimization
   - Could potentially run on a separate device


FUSION CANDIDATES (tt-lang opportunities)
------------------------------------------

1. AdaLN modulate (HIGH priority):
   Current: layer_norm -> multiply(scale+1) -> add(shift) = 3 ops
   Fused: single kernel reads hidden_states + shift/scale, writes modulated output
   Saves: 2 intermediate tensor allocations, 2 kernel launches
   Note: oasis has adaln_matmul_expand kernel that also fuses the temb MLP

2. RoPE + layout (HIGH priority):
   Current: matmul(q, perm_matrix) + multiply(q, cos) + multiply(q_swap, sin) + add
   = 4 ops per Q/K = 16 ops total (4 img/txt x Q/K)
   Fused: single kernel reads Q/K + cos/sin, does swap internally, writes result
   Saves: massive -- 16 kernel launches + intermediate tensors
   Note: oasis has rope_layout_kernel that also does SDPA layout transpose

3. Gated residual (MEDIUM priority):
   Current: multiply(gate, attn_out) + add(residual) = 2 ops
   Fused: single kernel reads residual + gate + value, writes sum
   Saves: 1 intermediate tensor, 1 kernel launch
   Occurs 4x per block (img/txt x attn/ffn)

4. SiLU + FFN GELU activation (LOW priority):
   ttnn.linear already supports activation="gelu" fusion
   SiLU for temb is a single call, not worth fusing

5. QKV projection fusion (MEDIUM priority):
   Current: 3 separate ttnn.linear calls for Q, K, V
   Could fuse into single matmul with concatenated weight matrix
   Saves: 2 kernel launches, better memory access pattern
   Would need to split output afterward


WEIGHT LOADING OPTIMIZATION
----------------------------

Current (170s): For each of 60 blocks, opens ALL safetensors shards,
scans ALL keys, loads matching ones. Extremely wasteful.

Fix implemented (not yet tested): extract_block_weights() pulls weights
directly from the already-loaded diffusers transformer module via
state_dict(). Should reduce to ~10-20s (just the host->device transfers).

Further optimizations:
  - Save pre-sharded bf16 weights to disk (skip float32 conversion)
  - Overlap CPU weight prep with device upload (pipeline the 60 blocks)
  - Keep process alive between generations (weights stay resident)


BUGS FOUND AND FIXED
---------------------

1. Missing RoPE in denoising loop:
   Warmup pass passed RoPE tables but denoising loop did not.
   Result: all-black images (no positional information).

2. AdaLN tile padding:
   temb [1,1,D] padded to [1,32,D] with zeros. After linear, only row 0
   had correct mod params. Concat-repeat tiled the broken pattern, so only
   every 32nd sequence position got correct modulation (PCC=0.29 vs ref).
   Fix: pre-expand temb to [1,32,D] with the correct value in all rows.

3. RoPE swap-weight ordering:
   Original approach: bake adjacent-element swap into QKV weight matrices.
   Problem: swap happened BEFORE QK-norm, but RoPE should be applied AFTER.
   rms_norm(swap(proj(x))) != swap(rms_norm(proj(x)))
   Fix: replaced with permutation matrix approach -- swap AFTER norm via
   Q @ P where P is a 128x128 adjacent-swap permutation matrix.
